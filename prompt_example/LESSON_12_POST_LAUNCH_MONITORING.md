# Lesson 12: 后发大监控 (Post-Launch Monitoring)

## 📚 核心理念

> **"系统上线只是开始，监控才是永远的工作"**

生产环境中，模型质量不是静态的。即使上线时精度 90%，也可能因为：
- 🔴 **数据分布变化** - 用户评论风格改变
- 🔴 **模型衰退** - 长期运行逐步下降  
- 🔴 **突发错误** - 新的错误模式出现

**Lesson 12 的解决方案：三层递进式监控**

```
┌─────────────────────────────────────────────────────────┐
│ 第 1 层: 日常样本审核 (Daily Review)                     │
│ 频率: 每天                                               │
│ 目的: 人工抽检，发现"模型没想到"的问题                   │
│ 告警: 问题率 > 10%                                       │
└─────────────────────────────────────────────────────────┘
           ↓
┌─────────────────────────────────────────────────────────┐
│ 第 2 层: 指标漂移检测 (Metric Drift Detection)           │
│ 频率: 实时 (每次评估后)                                  │
│ 目的: 自动检测精度下降                                   │
│ 告警: 指标低于阈值 或 环比下降 > 5%                     │
└─────────────────────────────────────────────────────────┘
           ↓
┌─────────────────────────────────────────────────────────┐
│ 第 3 层: 周报数据分析 (Weekly Analytics)                │
│ 频率: 每周                                               │
│ 目的: 趋势分析、根本原因分析、改进规划                   │
│ 输出: 周报 + 改进建议                                    │
└─────────────────────────────────────────────────────────┘
```

---

## 🎯 三层监控设计详解

### 第 1 层：日常样本审核 (Daily Review)

**目的**：人工是最后的防线。机器学习系统容易被"对抗性"用户评论迷惑。

**工作流程**
```
每天凌晨自动运行：
  1. 从昨日生产结果中随机抽取 N 条样本（N=5-10）
  2. 取出原始评论文本
  3. QA 人工审核（对标签、紧急度、行动等进行判断）
  4. 对比模型预测
  5. 记录异议项
```

**发现的问题类型**

| 问题类型 | 示例 | 严重程度 |
|---------|------|---------|
| **意图遗漏** | 评论提及"退款"但模型未打 refund_request | 🔴 高 |
| **紧急度错误** | 评论有加急符号但判为低紧急度 | 🔴 高 |
| **行动不足** | next_action 过于简短或无法执行 | 🟡 中 |
| **过度标签** | 不相关的标签被打上 | 🟡 中 |

**代码示例**
```python
# DailyReview.py
report = daily_sample_review(sample_size=5, seed=42)

# 输出：daily_review_2026-04-14.json
# {
#   "discrepancies_found": 2,
#   "discrepancies": [
#     {
#       "id": "cmt_015",
#       "issues": ["MISSED: refund_request", "URGENCY_MISMATCH"]
#     }
#   ]
# }
```

### 第 2 层：指标漂移检测 (Metric Drift Detection)

**目的**：自动告警，防止"无声的衰退"。

**监控指标**

```
关键指标                  | 基线    | 告警阈值
------------------------|---------|-----------
Intent Precision         | 0.89    | < 0.88, 下降 > 5%
Intent Recall           | 0.92    | < 0.92, 下降 > 5%
Intent F1               | 0.91    | < 0.90, 下降 > 5%
Urgency Accuracy        | 0.85    | < 0.88, 下降 > 5%
Next Action Pass Rate   | 1.00    | < 0.95, 下降 > 5%
```

**告警规则**

```
IF (当前指标 < 目标值) OR (相比前次下降 > 5%) THEN
  ALERT = TRUE
  记录到 metric_drift_YYYYMMDD.json
  发送告警通知
END
```

**代码示例**
```python
# MetricDriftDetection.py
report = detect_metric_drift(thresholds={
    "intent_f1": 0.90,
    "urgency_accuracy": 0.88
})

# 输出告警示例
# {
#   "alert_level": "WARNING",
#   "alerts": [
#     {
#       "metric": "urgency_accuracy",
#       "current": 0.80,
#       "threshold": 0.88,
#       "reasons": ["低于目标 (0.88)"]
#     }
#   ]
# }
```

**历史记录管理**

```
metric_history.json 存储最近 30 天的数据点：
[
  {"date": "2026-04-08", "intent_f1": 0.9057, ...},
  {"date": "2026-04-09", "intent_f1": 0.9045, ...},
  ...
]

这样可以进行周期性分析：
  - 周趋势
  - 月趋势
  - 异常检测
```

### 第 3 层：周报数据分析 (Weekly Analytics)

**目的**：看宏观趋势、找根本原因、规划改进。

**周报包含内容**

1. **周概览**
   - 审核样本数：20 条
   - 发现问题：5 项
   - 平均问题率：15%

2. **高频错误模式**
   ```
   遗漏意图标签      : 3 次 ← 最高频
   紧急度判断错误    : 2 次
   下一步行动不足    : 1 次
   ```

3. **指标周趋势**
   ```
   Intent F1: 0.9000 → 0.9057 ↑ +0.64%  (趋势向好)
   Urgency:   0.8500 → 0.8500 → (无变化,需关注)
   NextAction: 0.8000 → 1.0000 ↑ +25%   (改进显著)
   ```

4. **改进建议**（自动生成）
   ```
   1. 强化意图识别：遗漏模式频繁出现
      → 修改 MAIN_PROMPT 增强 refund_request 识别
   
   2. 紧急度规则优化：add("紧急", "加急", "急需")触发词
   
   3. A/B 测试：对比旧新 prompt 效果，验证改进
   ```

**代码示例**
```python
# WeeklyReport.py
report = generate_weekly_report()

# 输出：weekly_report_2026_W15.json
# {
#   "week": "2026-W15",
#   "samples_reviewed": 35,
#   "discrepancies_found": 5,
#   "error_patterns": {
#     "遗漏意图标签": 3,
#     "紧急度判断错误": 2
#   },
#   "recommendations": [
#     "强化意图识别：review MAIN_PROMPT...",
#     "优化紧急度规则：添加时间触发词..."
#   ]
# }
```

```

│                                                            │
│  日常审核       漂移检测        周报分析                  │
│  (样本问题)     (指标告警)      (趋势分析)                │
│      ↓              ↓               ↓                     │
│  ┌─────────────────────────────────────┐                 │
│  │        识别根本原因                  │                 │
│  │  (是 prompt 还是数据质量问题?)     │                 │
│  └─────────────────────────────────────┘                 │
│              ↓                                             │
│  ┌─────────────────────────────────────┐                 │
│  │      制定改进方案                    │                 │
│  │  (修改 prompt / 重标数据 / 调整规则) │                 │
│  └─────────────────────────────────────┘                 │
│              ↓                                             │
│  ┌─────────────────────────────────────┐                 │
│  │      A/B 测试验证                    │                 │
│  │  (新 prompt vs 旧 prompt)            │                 │
│  └─────────────────────────────────────┘                 │
│              ↓                                             │
│  ┌─────────────────────────────────────┐                 │
│  │      灰度上线新版本                  │                 │
│  │  (控制风险)                         │                 │
│  └─────────────────────────────────────┘                 │
│              ↓                                             │
│         【回到第一步，继续监控】                          │
│                                                            │
└──────────────────────────────────────────────────────────┘
```

### 完整的改进周期示例

**周一**
- 运行周报：发现"遗漏意图标签"频率最高
- 分析：prompt 中 refund_request 的触发词不完整

**周二-周三**
- 修改 MAIN_PROMPT：增强 refund_request 定义
- 创建新 PROMPT v2

**周四**
- A/B 测试：PROMPT v1 vs v2，对比 20 条样本
- 结果：v2 的意图 Recall 提升 3%

**周五**
- 灰度上线：v2 应用到 10% 的流量
- 监控 1 周，确认无问题

**下周一**
- 全量上线 v2
- 重新基线 Baseline（0.9057 → 新值）

## 💻 使用说明

### 快速开始

```bash
# 运行所有监控任务
cd /Users/xpeng/AI/github/BestPractice_AI/prompt_example

python MonitoringHub.py --run all

# 或仅运行单个任务
python MonitoringHub.py --run daily       # 日审核
python MonitoringHub.py --run drift       # 漂移检测
python MonitoringHub.py --run weekly      # 周报
```

### 日常使用计划

```
【每天】
  06:00 - 自动运行日审核 (cron: 0 6 * * *)
  输出: daily_review_YYYYMMDD.json

【每次评估后】
  自动运行漂移检测
  输出: metric_drift_YYYYMMDD.json

【每周一】
  08:00 - 自动生成周报 (cron: 0 8 * * 1)
  输出: weekly_report_YYYY_WXX.json
  
  10:00 - 周报评审会议
  讨论: 本周发现的问题 + 下周改进计划
```

### 告警响应流程

```
IF 漂移检测发现 Alert THEN:
  1. Slack 消息: "@team Metric Alert: F1 down 6%"
  2. 立即查看 metric_drift_YYYYMMDD.json
  3. 对应查阅 daily_review_* 找具体问题
  4. 开启应急讨论
  5. 准备改进方案
ENDIF
```

## 🎓 Lesson 12 核心要点总结

| 层级 | 任务 | 频率 | 目的 | 告警 |
|-----|------|------|------|------|
| L1 | 日常审核 | 每天 | 发现新错误模式 | 问题率 > 10% |
| L2 | 漂移检测 | 实时 | 自动监测精度 | 指标下降 > 5% |
| L3 | 周报分析 | 每周 | 趋势分析 + 规划 | 频繁告警 |

**核心理念：** 
> 监控是"持续改进"的入口。通过三层递进式监控，你可以尽早发现问题、快速定位原因、持续优化系统。
